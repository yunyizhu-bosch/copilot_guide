# GitHub Copilot 指导

## 关于本项目

本项目是一个数据分析框架，旨在通过结合结构化知识库和 AI 编程，实现对业务需求的快速响应和自动化分析。

## 核心工作流
1. **理解业务需求**: 用户在 `.py` 或 `.ipynb` 文件中用自然语言（注释形式）描述一个业务问题。需求可能只使用数据字典中的常规数据，也可能同时引入本地临时文件。
2. **参考知识库**: 你必须首先查阅 `.github/assistant/` 目录下的知识文件，识别常规数据的表结构、字段含义和典型业务场景。
3. **识别数据来源**: 明确本次分析使用的是常规数据、临时文件，还是二者结合。对于临时文件，优先使用用户在需求中提供的字段含义作为本次分析的临时数据字典。
4. **需求答疑**: 如果对于需求或者数据有疑问，先请用户澄清。尤其是当临时文件的字段含义没有说明清楚时，必须先提问，不要自行假设。如果发现关键的数据缺失时，也要向用户确认。
5. **数据探索 (EDA)**: 在读取数据后，**你必须主动执行探索性分析**，例如使用 `.info()`, `.describe()`, `.shape`, `.value_counts()` 等方法来理解数据概况、检查数据质量。**不要等待用户指令**。对于混合分析，既要探索临时文件，也要核对它与常规数据在关联键、字段格式、取值范围上的兼容性。如果发现有关键的数据问题且你无法自行解决，可以请用户解决。
6. **制定分析计划**: 基于用户输入和**数据探索 (EDA)**的结果制定分析计划，用注释形式写出清晰、可执行的分析步骤。如果探索中发现数据问题（如缺失值、异常值、关联键不一致、数据类型不一致等），计划中必须包含相应的处理步骤。
7. **生成代码**: 根据分析计划、知识库和用户提供的临时字段说明，生成 Python 代码。代码规范包括
7.1. 代码必须清晰、规范，并且包含必要的注释来解释每一步的目的和逻辑。
7.2. 尽量使用 pandas 的 groupby、merge 等方法来实现数据的清洗、转换和分析，尽量避免使用 for 循环。
7.3. 变量的命名要具有描述性，能够反映变量的内容和用途。
7.4. 如果需要从数据库中读取数据，注意查询的效率，尽量只查询必要的字段和数据范围。
8. **执行与迭代**: 执行代码并检查结果。如果结果不符合预期，或者代码报错，请重新检查数据和分析计划，找出问题所在并修正。如果无法自行解决，可以请用户解决。
9. **结果检查与自动纠偏**: 产出结果后，必须先自动列出检查点并执行检查，再决定是否交付结果。检查示例：
9.1. 结果完整性检查（记录数、关键字段非空、分组后汇总是否缺失）。
9.2. 口径一致性检查（过滤条件、时间范围、分母分子定义是否与需求一致）。
9.3. 数值合理性检查（最小/最大值、占比范围、环比同比异常波动、重复计数风险）。
9.4. 关联质量检查（join 前后行数变化、主键唯一性、匹配率、是否出现意外笛卡尔积）。
9.5. 与历史或常识对比检查（如可获得历史结果，则比较趋势是否异常）。
9.6. 挑几个结果样例，查看是否能根据这些样例的相关原始数据推断出结果。
10. **异常处理与重跑**: 如果检查结果不合理，必须定位问题代码并修复后重跑；必要时回退到 EDA 和分析计划阶段重新确认。每次修复都要记录“问题现象 → 根因定位 → 修改内容 → 重跑结果”。
11. **结果汇报与用户补充**: 向用户明确汇报“检查点清单、每项检查结论、未通过项及修复动作、最终是否通过”。同时显式列出“未覆盖但建议补充的检查点”，并邀请用户补充业务特定检查规则。
12. **更新知识库**: 如果在分析过程中发现知识库中的数据定义、业务场景或者代码模板有误或者可以优化，必须向用户反馈，并在得到确认后更新 `.github/assistant/` 目录下的相关文件。
